**Context & Goal:**
You are an expert Frontend Developer. Build a modern, responsive, dark-mode Single Page Application (SPA) called "MDM Location Data Generator", using a standard 3-column dashboard layout. 
Use React + Vite and Tailwind CSS. This tool will be used by QA testers to dynamically generate synthetic test data payloads (JSON and XML) for DaVita's Location Master Data Management (MDM) system, validate them against the DB, post them, and query Kafka queues.

**Critical Setup - CORS Fix via Vite Proxy (or Mock if building in Claude Artifacts):**
If building in a local IDE, configure a proxy in `vite.config.js` to bypass CORS for `https://mdmace-{env}.davita.com`, `https://intgkafka-{env}-rest-apis.int-svcs-np.davita.com`, and `https://mdm-{env}.davita.com:443`. The frontend must *never* fetch the external URLs directly. (If building as a Claude web artifact, strictly mock these network responses with a `setTimeout`).

---

### **UI Layout Requirements (3 Columns):**

**1. Left Column: Data Generator Control**
*   **Environment Selector:** Radio buttons or toggles for `dev`, `qa`, `uat`. (This updates all target API URLs dynamically).
*   **Domain Selectors (Radio/Dropdown):**
    *   **Legal Entity** (Sub-toggle: Domestic, International) -> `/Location/IS06/legalEntityMDMProcess`
    *   **Org Assignment** (Sub-toggle: Palmer, Division, Group, Region) -> `/Location/IS06/orgAssignment`
    *   **Facility Hierarchy** (Sub-toggle: OPER, SG&A) -> `/Location/IS06/facilityHierarchy`
    *   **Facility Master** (Sub-toggle: Domestic, International. PLUS a "Prefix" dropdown: None/Standard, ACQ-, CLSD-, CNXL-, DNVO-, HOLD-, SOLD-, TERM-, JV DNVO-, DIVEST-, DARK-, INTL) -> `/Location/IS06/facilityMasterMDMProcess`
    *   **Karma** -> `/Location/IS06/locationKarmaLoad`
    *   **MNA** -> `https://mdm-{env}.davita.com:443/MDMWSProvider/MDMService`
    *   **NSHN** -> `https://mdm-{env}.davita.com:443/MDMWSProvider/MDMService`
*   **Endpoint URL:** A read-only text input showing the proxy-resolved target URL.

**2. Middle Column: Generated Payloads**
*   **Generate Button:** Prominent button to build the Payload.
*   **Output Display:** A large syntax-highlighted `<textarea>` or `<pre>` block displaying the dynamically generated JSON or XML. User can manually edit it.
*   **Action Buttons:** 'Copy to Clipboard' and 'Run Payload' (Executes a real `POST fetch()`—use `application/json` for JSON domains, and `text/xml` for MNA/NSHN domains).
*   **Response Panel:** A terminal-like box below the payload showing HTTP status codes, server responses, or network errors.

**3. Right Column: Data Validation Control**
*   **End to End Validation:** Button to simulate generation, extract EUID, and query queues.
*   **Preprocess Queue:** Input field for EUID and a "Query Queue" button. Makes a `GET` request to: `/data/topic/mdm.facilities.preprocess/count/10` (via proxy). Parses the list and finds the matching EUID.
*   **Event Queue:** Input field for EUID and a "Query" button. Makes a `GET` request to: `/data/topic/mdm.facilities.events/count/2` (via proxy).
*   **Query Location By EUID:** Input field for EUID, hitting `/Location/IS01/getLocationDetailsByEUID?EUID=<EUID>` (via proxy).

---

### **Global Generation Rules:**
*   **Dates:** `YYYY-MM-DD`. Date-times: `YYYY-MM-DD HH:mm:ss.ffffff` (or `.0` for XML).
*   **meta.id / requestID:** Random 19-digit number.
*   **State Persistence:** If Legal Entity is generated, store its ID globally. If Hierarchy is generated, store its ID globally.
